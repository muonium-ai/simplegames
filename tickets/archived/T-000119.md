## Goal
Make minesweeper's autoplay actually play instead of cheating, ship
deduction- and CSP-based solvers, and align outcome telemetry across
both minesweeper entry points.

## Acceptance Criteria
- [x] `minesweeper_with_probability.py --autoplay` plays step-by-step
      visibly (no instant solve_it cheat).
- [x] `minesweeper_with_solver.py --solver csp_solver` runs end-to-end
      without "No moves available" loops.
- [x] Both scripts print `<WIN|LOSS> in X.XXs — flagged N/99, opened M/381`.
- [x] At least one new solver wins materially more than `random_solver`
      on Expert (30x16, 99 mines); csp_solver benchmarks at ~42-51%
      across N=100/500 samples.

## Notes
- The original `--autoplay` called `solve_it()` once, which paints the
  answer in a single tick. Replaced with `autoplay_step()` driven by
  direct constraint deduction; the existing `update_probabilities()`
  uses `max(...)` and silently drops "definitely safe" constraints, so
  it stays only for the on-screen probability labels.
- `handle_game_over` / `reveal_all_mines` now preserve FLAGGED state so
  flag counts survive the post-mortem and the outcome line credits
  correctly-flagged mines.
- Three new solvers under `minesweeper/solvers/`:
  - `basic_solver`: single-cell deduction (~5% wins on Expert).
  - `subset_solver`: + subset-difference (catches 1-2-1, 1-1; ~27%).
  - `csp_solver`: connected-component enumeration with constraint
    propagation, MAX_CONFIGS=50_000 per component (~42-51%).
- Subtle gotcha: running the framework as a script makes its module
  `__main__`; `from minesweeper_with_solver import CellState` then
  re-imports the file as a second module with a fresh Enum class, and
  `cell.state == CellState.HIDDEN` always returns False. Solvers
  derive CellState lazily from `type(game.grid[0][0].state)` on the
  first `next_move()` call to dodge this.
