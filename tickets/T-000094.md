## Goal
Define a uniform multi-level convention so the Tier 1 games (bricks, flappybird, tetris, 2048, minesweeper, sudoku) share one mental model for what "levels" mean and how the player interacts with them.

## Acceptance Criteria
- [ ] **Level selection**: each game shows level options on its start screen (or supports `--level <n>` argv where a start screen would be overkill). Default selection = level 1 (easiest).
- [ ] **HUD display**: current level is visible in the HUD during gameplay (top-left or near the score). Format: `Level: N` or `Level: <name>` if levels have names (e.g., Easy/Medium/Hard).
- [ ] **Win condition**: each level has a clear win condition. Reaching it either auto-advances to the next level or returns to a "level complete" screen with a clear "next level" affordance.
- [ ] **Level-up feedback**: when a level is cleared, show a brief (~1 second) overlay announcing the next level. Use the non-blocking deadline-loop pattern from T-000056 so ESC still works.
- [ ] **Persistence (optional)**: highest level reached is saved to `<game>/saved_progress.json` (script-relative per T-000055) and loaded on startup; level select highlights the highest unlocked level. Saved-progress files MUST be gitignored.

## Notes
- ESC handling from T-000058..T-000074 is preserved everywhere — never bind ESC to a level action.
- Where a game already has an implicit level system (tetris ramping `fall_speed` with `lines_cleared_total`; 2048 milestone announcements), the per-game ticket should *surface* it as a real level concept rather than re-invent.
- Level select UI should match each game's existing visual style — don't introduce a new HUD framework.
- For games where a "level" is really a difficulty mode (minesweeper Beginner/Intermediate/Expert, sudoku Easy/Medium/Hard), use the named-difficulty form. For games with sequential progression (bricks, flappybird), use numeric levels.
- Tier 2 games (spaceinvaders, snake, pong, mazes, asteroid, rubiks_cube) get a separate spec + tickets later.
- Out of scope: Tier 3 games (gameoflife, genetic_car, chess, card_games).
- This is the **spec only**. Per-game implementation tickets reference this one via `depends_on`.
